**Step 6: Add known landmine about CONFIG_DATA vs load_config()**

Add to the `## 6. Known landmines` section:

```
- **`get_live_prediction` reads per-model config from `CONFIG_DATA`, not `load_config()`**
  The live prediction endpoint uses `CONFIG_DATA.get(f"{role}_provider")` directly rather than
  `load_config().get(...)`. This ensures settings panel `PATCH /settings` updates take effect
  immediately. The existing `get_ai_prediction` endpoint uses `load_config()` for some reads
  and `CONFIG_DATA` for others — both work when a config file is active, but in tests you
  must mutate `CONFIG_DATA`, not mock `load_config()`.
- **`predict()` now accepts `temperature=0.7`** — all 4 provider adapters and the `predict_llm()`
  wrapper accept this parameter. The default is 0.7 (was hardcoded 0.0).
- **`validate_per_model_config()`** — new validation function in `services/llm_providers.py`.
  Checks per-model config completeness. Called by `get_live_prediction()`.
```

## Risks and open questions

1. **Doc drift is silent.** There's no automated check that docs match the code. The best defense is reading the actual source while updating each doc. Each task above includes the specific lines/sections to change — don't guess.

2. **cli.py and main.py `load_config()` are out of sync on the `theme` key.** cli.py's defaults don't include `"theme": "dark"` (main.py line 116 has it). This is a pre-existing discrepancy, not introduced by Phase 4. Don't fix it in this task.

3. **The cli.py config dict has no trailing comma on its last line.** Adding new keys after it requires adding a comma to the existing last line (`"helper_agent_prompt_template": CLI_DEFAULT_CHAT_TEMPLATE` → add `,` then the new keys). This is handled correctly in the diff above.

4. **testcases.md Tier 9 tests are manual (browser).** They cannot be automated without browser automation (Playwright/Cypress). The "—" verdict column indicates they need manual walkthrough.
