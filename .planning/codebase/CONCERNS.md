## 1. Technical Debt

### 1.1 `data.py` import staleness ‚Äî by-value `import app.config as cfg`

**File:** `app/data.py` (lines 7, 12, 16, 21, 160)
**Problem:** `data.py` does `import app.config as cfg` and reads `cfg.DATA_FILE_PATH` / `cfg.DATA_FILE_TYPE` at call time. While Python re-resolves module attributes on each access (module-level attributes _are_ re-read), **assignments to the bare name at module scope** (e.g. `from app.config import DATA_FILE_PATH`) would create a by-value local copy that never updates.

The current `import app.config as cfg ‚Üí cfg.DATA_FILE_PATH` pattern actually works correctly because it resolves through the module object. However, the `from app.config import *` in `main.py` (line 7) does create by-value copies of all names in the module namespace of `main.py`. Any external code (e.g. `cli.py`, tests) that does `from main import CONFIG_DATA` gets a reference to the **same dict object** (since `CONFIG_DATA` is a mutable dict), so mutations _do_ propagate ‚Äî but the bare string values (`DATA_FILE_PATH`, etc.) are by-value copies that **would not update** if reassigned in `config.py` after import.

**Risk:** If any module does `from app.config import DATA_FILE_PATH` (not via the module reference), it gets a stale string copy. This has already bitten `main.py` (line 42 uses `DATA_FILE_PATH` which is a by-value copy of the original value from config.py at import time). `main.py` line 7's `from app.config import *` is the primary vector.

### 1.2 Module-level mutable state in `config.py`

**File:** `app/config.py` (lines 7‚Äì22, 110‚Äì113)
**Problem:** Global mutable variables (`DATA_FILE_PATH`, `DATA_FILE_TYPE`, `CONFIG_PATH`, `CONFIG_DATA`, `AUTO_POSITIONS`) are defined at module scope and mutated by `set_data_file()`, `set_config()`, `load_config()`. The module body has **side effects at import time** (lines 13‚Äì22: reading config file, setting `AUTO_POSITIONS`). This makes the module stateful and order-dependent.

**Evidence:**
- `config.py` line 13‚Äì14 runs `os.environ.get('ABSA_CONFIG_PATH')` at import time
- `config.py` line 22: `AUTO_POSITIONS = CONFIG_DATA.get('auto_positions', False)` runs at import time
- `settings.py` line 83‚Äì93: `PATCH /settings` mutates the global `CONFIG_DATA` dict directly and writes to a file

**Risk:** Tests cannot safely import `config.py` in isolation. Concurrent requests to `PATCH /settings` race on the same mutable dict.

### 1.3 `main.py` re-exports for backward compatibility

**File:** `main.py` (lines 6‚Äì9)
**Problem:** `main.py` re-exports `from app.config import *` and selective functions from `app.data` / `app.positions` solely for backward compat with `cli.py` and tests that `from main import CONFIG_DATA`. This creates a circular dependency smell and masks the real module boundaries.

**Evidence:**
- `main.py` line 6: `# Re-export for backward compat (cli.py, tests mutate main.CONFIG_DATA etc.)`
- `cli/config.py` line 9: imports from `services.prediction` directly (correct)
- `cli/__init__.py` line 500: imports from `services.llm_providers` directly (correct)
- But the docstring in `main.py` admits `cli.py, tests mutate main.CONFIG_DATA`

### 1.4 Duplicate default values across config layer

**Problem:** The same default config values are hardcoded in:
- `app/config.py` `load_config()` (lines 60‚Äì107)
- `cli/config.py` `ABSAAnnotatorConfig.__init__()` (lines 18‚Äì67)
- `frontend/src/App.tsx` `DEFAULT_SETTINGS` (lines 78‚Äì99)

If a new config key is added, it must be added in 3+ places. There is no single source of truth.

### 1.5 Mixed argument naming conventions in `cli/__init__.py`

**File:** `cli/__init__.py`
**Problem:** Some CLI args use `--snake-case` (e.g. `--disable_implicit_opinion`, `--disable_clean_phrases`) while most use `--kebab-case` (the standard). This is inconsistent.

**Evidence:**
- Line 92: `--disable_implicit_opinion` (snake_case)
- Line 99: `--disable_clean_phrases` (snake_case)
- Every other flag uses kebab-case

## 2. Security

### 2.1 CORS allows all origins

**File:** `main.py` (lines 21‚Äì27)
**Evidence:**
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # ‚Üê allows ANY website to call this API
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
**Risk:** Any website visited by a user who has AnnoABSA running on localhost:8000 can make authenticated (origin-based) requests. The `allow_credentials=True` + `allow_origins=["*"]` combination is explicitly warned against in the FastAPI/CORS docs ‚Äî it cannot be safely combined.

### 2.2 No authentication

**Problem:** All endpoints are public. There is no auth middleware, no API key validation on HTTP requests, no session management. The `session_id` is an arbitrary string passed via CLI args ‚Äî it's not validated anywhere.

**Files:** All route files under `app/routes/` ‚Äî none have any auth checks.

### 2.3 File upload with no content validation

**File:** `app/routes/upload.py` (lines 22‚Äì33)
**Evidence:**
- Only checks file extension (`.csv`, `.json`)
- Does NOT validate: file size, MIME type, file content structure before saving
- Does NOT scan for malicious content
- Saves files to `app/uploads/` directory with predictable names (`uploaded_{timestamp}.csv`)

**Risk:** A user could upload a very large file to cause DoS, or a file with embedded scripts.

### 2.4 API keys exposed via CLI args

**Files:** `cli/__init__.py` (lines 176, 228‚Äì231)
**Evidence:**
```python
parser.add_argument("--openai-key", metavar="API_KEY", ...)
parser.add_argument("--anthropic-key", metavar="API_KEY", ...)
```
**Risk:** API keys passed as CLI arguments are visible in process listings (`ps aux`), shell history, and potentially system logs. Should use environment variables or a config file.

### 2.5 API keys stored in config JSON on disk

**File:** `cli/runner.py` (lines 96‚Äì98)
**Evidence:** `temp/temp_absa_config.json` is written with full API keys in plaintext.
**Risk:** If `temp/` is committed or left on disk, API keys leak.

## 3. Performance

### 3.1 BERT model ~1.2 GB ‚Äî first-load latency

**File:** `services/nlp_helpers.py` (lines 30‚Äì38)
**Evidence:**
```python
_sentiment_classifier = pipeline(
    "sentiment-analysis",
    model="savasy/bert-base-turkish-sentiment-cased",
)
```
The `savasy/bert-base-turkish-sentiment-cased` model is ~413 MB (BERT-base). Combined with `intfloat/multilingual-e5-small` (~470 MB) and NlpToolkit dictionaries, total first-load is ~1.3 GB.

**Problem:** Downloads on first request to `/nlp/sentiment` or `/nlp/embedding-similarity`. The first request can take 30‚Äì120 seconds while models download to `~/.cache/huggingface/`. Even after caching, model loading into GPU/CPU memory takes several seconds.

**Acknowledged in:** `setup.sh` line 94‚Äì96 and `setup.bat` line 110‚Äì111.

### 3.2 TF-IDF re-fits on every active learning scan

**File:** `services/active_learning.py` (lines 30‚Äì33, 145‚Äì149)
`app/routes/learning.py` (lines 47, 60‚Äì61, 74‚Äì75)

**Problem:** Every call to `GET /learning/suggestions` and `GET /learning/predict/{idx}` calls `train_labeled_data()` which **re-fits** the entire `TfidfVectorizer + LogisticRegression` pipeline from scratch. There is no model caching or incremental update.

**Evidence:**
- `learning.py` line 60: `model_data = train_labeled_data(texts, label_sets)` ‚Äî called on every request
- `active_learning.py` lines 145‚Äì149: `model.fit(x_labeled, y_labeled)` ‚Äî full re-fit each time

**Impact:** O(n) per request where n = labeled reviews. With thousands of labeled reviews and frequent active learning queries, this becomes a bottleneck.

### 3.3 BM25 re-indexes on every agent chat

**File:** `services/prediction.py` (lines 194‚Äì198)
`app/routes/reviews.py` (lines 163‚Äì164)

**Problem:** `get_most_similar_examples()` creates a new `BM25Okapi` instance from scratch on every call. The BM25 index is built from tokenized examples, which means the entire example list is re-tokenized and re-indexed on every `/agent/chat` request.

**Evidence:**
- `prediction.py` line 198: `bm25 = BM25Okapi(tokenized_examples)` ‚Äî new instance every call
- `reviews.py` line 164: `get_most_similar_examples(req.review_text, examples, n=2)` ‚Äî called on every chat message

### 3.4 No database ‚Äî in-memory from CSV/JSON

**Problem:** The entire "database" is reading/writing CSV or JSON files on every request. Every `load_data()` call reads the entire file into memory. Every `save_data()` call writes the entire file back.

**Files:** `app/data.py`, all route files
**Evidence:**
- `data.py` line 12: reads entire file
- `data.py` line 22: writes entire file
- Routes call `load_data()` + `save_data()` multiple times per request (e.g., `reviews.py` lines 101‚Äì113: load, mutate, save = 2 file I/O ops per save)

**Impact:** O(n) memory per request, file I/O as the bottleneck. Cannot scale beyond a few thousand reviews. No indexing, no querying, no transactions.

## 5. Compatibility

### 5.1 `setuptools < 75` pin

**Files:** `pyproject.toml` (line 22), `setup.sh` (line 41), `setup.bat` (line 43)
**Evidence:**
```toml
"setuptools<75",
```
**Problem:** The `setuptools<75` pin is required by `nlptoolkit-*` packages (StarlangSoftware NLP toolkit), which depend on legacy setuptools behavior removed in setuptools 75+. This pin is a dependency constraint, not a project concern per se, but it blocks users who need newer setuptools for other projects from using the same virtualenv. It's listed as a runtime dependency in `pyproject.toml`, which means it gets installed with `pip install -e .` ‚Äî but it's a `pip` tool constraint, not a library dependency.

### 5.2 React 19 + `@testing-library/react` incompatibility

**File:** `frontend/package.json` (lines 8, 19)
**Evidence:**
- React `^19.1.1`
- `@testing-library/react` `^16.3.2`

**Problem:** React 19.1.1 CJS build does not export `React.act`, which `@testing-library/react` v16.3.2 calls internally. This causes test crashes. Documented in multiple test files:

**Evidence from test files:**
- `frontend/src/components/HelperAgentChatbox.test.tsx` line 3: `// NOT using @testing-library/react due to React 19.2.7 CJS bug`
- `ModelTripletColumn.test.tsx` line 3: same
- `NlpHelperToolbar.test.tsx` line 3: same
- `SettingsPanel.test.tsx` line 3: same
- `agentdocs/phase3/phase3_completion_report.md` lines 128, 242‚Äì243: documents the bug explicitly

**Impact:** Component tests use manual `ReactDOM.createRoot` + `flushSync` workarounds instead of `render()` from testing-library.

### 5.3 Root `package.json` vite pin

**File:** `package.json` (root, line 12)
**Evidence:**
```json
"vite": "^8.1.4"
```
**Problem:** This is in the root `package.json` but the actual frontend is in `frontend/`. The root `package.json` has `vite` as a devDependency but also uses `concurrently`. The root and `frontend/` have different vite versions (`^8.1.4` vs `^4.5.14`). This can cause confusion about which vite is running.

### 5.4 git-bash / MSYS environment

**Problem:** The project is developed on Windows with git-bash/MSYS. The `setup.bat` is the Windows-native bootstrap, but the root `package.json` uses `set PYTHONIOENCODING=utf-8` (Windows `set`, not bash `export`). This means the `start:backend` script only works on Windows CMD, not in git-bash or Linux.

**File:** `package.json` (line 5)
```json
"start:backend": "set PYTHONIOENCODING=utf-8 && uvicorn main:app --reload --port 8000"
```
On Linux/macOS this would need `PYTHONIOENCODING=utf-8 uvicorn ...`. The `setup.sh` works on Linux/macOS but the root `package.json` scripts assume Windows.

## 6. Missing Features

### 6.1 No persistent database

**Files:** `app/data.py`
**Problem:** All data is stored in CSV or JSON files. There is no database layer ‚Äî no SQLite, no PostgreSQL, no ORM.

**Impact:**
- No concurrent write safety (two simultaneous saves can corrupt the file)
- No indexing ‚Äî searching requires full file scan
- No transactions ‚Äî partial writes on crash
- No schema enforcement
- No migration capability

### 6.2 No multi-user support

**Problem:** The application has no concept of users. The `session_id` is cosmetic (an arbitrary string logged but never validated). Multiple users annotating the same data would overwrite each other's work.

**Evidence:**
- No user model
- No auth middleware
- `session_id` in config is purely informational
- Single `CONFIG_DATA` global means one configuration for everyone

### 6.3 No undo functionality

**Problem:** There is no undo for annotation actions. Once a triplet is saved via `handleNextReview()` ‚Üí `POST /review/{idx}/save`, the data is written to disk immediately. There is no:
- Undo/redo stack
- Soft-delete / versioning
- Confirmation dialog before save
- "Last N saves" rollback

**Evidence:** `app/routes/reviews.py` lines 97‚Äì131: `save_review_triplets()` writes directly to file with no rollback mechanism.

### 6.4 Active learning requires 2+ annotated reviews

**Files:** `services/active_learning.py` (line 139), `app/routes/learning.py` (lines 130‚Äì134)
**Evidence:**
```python
if n_labeled < 2:
    return None  # No model trained
```
and
```python
if len(labeled_texts) < 2:
    raise HTTPException(status_code=400, detail="...need at least 2 to train.")
```
**Impact:** Active learning is unavailable until at least 2 reviews are annotated. The first few reviews must be annotated blindly (no ML guidance).
